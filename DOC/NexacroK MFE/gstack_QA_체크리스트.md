# gstack 기능 QA(기능 테스트) 체크리스트

gstack(https://github.com/garrytan/gstack)의 `/qa`, `/qa-only` 스킬을 활용한 기능 테스트 체크리스트입니다.
- `/qa <URL>`: 버그 발견 시 실제 코드 수정 + 회귀 테스트 자동 생성까지 수행
- `/qa-only <URL>`: 동일한 방식으로 테스트하되 리포트만 작성 (코드 수정 없음)

## 1. 사전 준비

- [ ] gstack 설치 확인 (`~/.claude/skills/gstack` 존재, `./setup` 완료)
- [ ] 테스트 대상 URL 확보 (staging 또는 production)
- [ ] 로그인이 필요한 경우 `/setup-browser-cookies`로 실제 브라우저 쿠키 임포트
- [ ] 배포 관련 테스트가 필요하면 `/setup-deploy` 사전 설정 확인
- [ ] 직전 `/plan-eng-review`에서 작성된 테스트 매트릭스가 있는지 확인 (있으면 `/qa`가 이를 참고)

## 2. 실행

- [ ] `/qa <URL>` 또는 `/qa-only <URL>` 실행
- [ ] 실제 브라우저(Chromium)가 정상적으로 열리는지 확인
- [ ] 에이전트가 주요 플로우를 순회하며 클릭/입력을 수행하는지 확인
- [ ] 캡차/2단계 인증 등으로 막히면 `$B handoff`로 수동 처리 후 `$B resume`

## 3. 기능 테스트 항목

### 인증/계정

- [ ] 회원가입

- [ ] 권한(역할)별 접근 제어

- [ ] 뒤로가기·새로고침 시 상태 유지
- [ ] 404 및 에러 페이지 처리

### 폼 / 입력
- [ ] 필수값 검증
- [ ] 잘못된 입력값에 대한 에러 메시지
- [ ] 제출 성공/실패 피드백

### 데이터 처리 (CRUD)
- [ ] 생성 / 조회 / 수정 / 삭제 동작

## 4. MFE(마이크로프론트엔드) 테스트 방법

gstack의 `/qa`, `/browse`는 코드가 아니라 **실제 렌더링된 브라우저 화면**을 대상으로 동작합니다.
따라서 여러 개의 MFE(remote 앱)가 모듈 페더레이션·single-spa·iframe 등으로 조립되어 있어도
합쳐진 결과물이 뜨는 **호스트(컨테이너) URL 하나**만 있으면 그대로 `/qa`, `/browse`를 쓸 수 있습니다.
다만 MFE 특성상 아래 항목을 추가로 챙겨야 합니다.

### 사전 준비
- [ ] 테스트할 호스트(컨테이너) URL 확정 (여러 remote가 합쳐져 뜨는 진입 URL)
- [ ] 개별 MFE가 서로 다른 서브도메인/포트를 쓰는 경우, 해당 도메인 모두 파악
- [ ] 인증이 도메인마다 분리되어 있으면 `/setup-browser-cookies`에서 관련 도메인 전부 선택해서 임포트
- [ ] 특정 MFE 하나만 새로 배포된 경우, 그 MFE의 브랜치 diff를 기준으로 `/qa`를 돌릴지(diff-aware 모드) 전체 호스트를 돌릴지 결정

### 실행 방법
- [ ] 빠른 점검: `/qa <호스트URL> --quick` (홈 + 주요 내비게이션 5개, 30초 스모크 테스트)
- [ ] 전체 점검: `/qa <호스트URL>` (전체 탐색, 5~15분, 5~10개 이슈 리포트)
- [ ] 배포 전후 비교: `/qa <호스트URL> --regression baseline.json` (이전 baseline과 diff)
- [ ] 특정 MFE 저장소 안에서 작업 중이면 `/qa`만 실행 시 git diff 기반으로 영향받는 라우트만 자동 테스트

### MFE 특화 체크 항목
- [ ] **모듈 로드 실패** — `remoteEntry.js` 등 원격 청크 로드 실패를 네트워크 요청(`$B network`)에서 4xx/5xx·timeout으로 확인
- [ ] **공유 의존성 충돌** — React/상태관리 라이브러리 singleton 버전 불일치로 인한 콘솔 에러(`$B console --errors`)
- [ ] **MFE 간 라우팅 전환** — 한 remote에서 다른 remote로 이동할 때 상태·인증·쿼리스트링 유지 여부
- [ ] **독립 배포 영향 범위** — 한 MFE만 새로 배포됐을 때 다른 MFE(호스트 포함)가 깨지지 않는지
- [ ] **iframe 기반 MFE** — 프레임 안에 렌더링되는 구조면 `$B frame --url <pattern>`으로 해당 프레임으로 전환해 내부 요소 검증 후 `$B frame main`으로 복귀
- [ ] **다중 탭/윈도우 MFE** — 여러 MFE가 별도 탭으로 열리는 구조면 `$B tabs`로 목록 확인, `$B tab-each "text"` 등으로 전체 탭 일괄 점검
- [ ] **공통 헤더/네비/디자인시스템** — 여러 MFE에 걸쳐 공유되는 컴포넌트가 각 MFE에서 동일하게 동작하는지

### 배포 후
- [ ] 해당 MFE만 배포된 경우에도 `/canary <호스트URL>`로 합쳐진 전체 화면 기준 모니터링 (다른 MFE와의 통합 회귀 감지)
- [ ] `/benchmark <호스트URL>`로 배포 전후 로드 시간·Core Web Vitals 비교 (다른 remote 로딩 지연 영향 포함)

## 5. 테스트 후 확인

- [ ] 발견된 버그 목록 정리 (`/qa-only`는 리포트, `/qa`는 커밋 로그 확인)
- [ ] `/qa`로 수정된 건별 회귀 테스트 자동 생성 여부 확인
- [ ] 수정 사항에 대한 `/review` 재검토 (선택)
- [ ] 배포 후 모니터링이 필요하면 `/canary` 실행
- [ ] `/ship` 전 테스트 커버리지 확인
